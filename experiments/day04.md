# Day 4: The Runner & JSON Validation 

## What I Built Today
Today, I built **The Runner**, a fully automated testing engine that executes the experiment plans generated by the Day 3 Planner. 
* **Dynamic Execution:** Instead of hardcoding prompts, the Runner reads the JSON plan, maps the required models to their exact API clients (Groq/NVIDIA), and orchestrates the tests.
* **Batch Processing:** It loops through a dataset of 20 customer support emails, safely replacing the `{{input}}` placeholder with real data, resulting in 60 automated API calls.
* **Data Aggregation:** The engine automatically tracks latency and JSON parsing success, exporting everything to a structured Pandas DataFrame and saving it as `day04_results.csv`.
* **Automated Scoring:** I implemented `acc_scorer.py`, which uses Python's `ast.literal_eval` to safely parse the AI output and cross-references it against a dictionary of human-labeled Ground Truth data to calculate the exact accuracy percentage of each model.

## What Errors I Faced & How I Fixed Them

### Error 1: The "0/60 JSON" Failure (The Human Prose Bug)
* **The Problem:** My initial run resulted in a 0% JSON success rate across all models. The models were answering like helpful humans (e.g., *"This email is urgent because..."*). 
* **Root Cause:** The Day 3 Planner generated a prompt (`"Classify this email...: {{input}}"`) but failed to include explicit instructions to output structured JSON data. 
* **The Fix:** I updated the `updated_planner.py` meta-prompt to strictly command Llama-70B to append JSON instructions. I also built a `wrap_for_json` safety net in `runner.py` that forcefully attaches a strict visual JSON schema (e.g., `{"label": "...", "confidence": 0.0-1.0}`) to the prompt before it is sent to the models.

### Error 2: Qwen-32B's Token Guillotine (Leaking `<think>` Tags)
* **The Problem:** Even with the wrapper, Qwen-32B failed on complex samples, returning a `No JSON found` error with raw output containing unfinished `<think>` tags.
* **Root Cause:** Qwen is a reasoning model that thinks out loud. The `max_tokens` parameter was set to 300, which violently cut the model off mid-thought before it could ever write the final JSON object.
* **The Fix:** I increased `max_tokens` from 300 to 800, giving Qwen enough runway to finish its internal reasoning and output the JSON block, resulting in a perfect 20/20 parse rate.

### Error 3: Llama-8B's Refusal (The Wrapper Bypass)
* **The Problem:** Llama-8B continued to output plain text instead of JSON on a few samples, causing parser crashes.
* **Root Cause:** My original wrapper had a logic flaw (`if "json" in task_prompt: return task_prompt`). Because the prompt contained the word "JSON", the script bypassed attaching the strict visual schema. Without that explicit visual structure, the smaller 8B model panicked and defaulted to prose.
* **The Fix:** I removed the bypass logic, ensuring the strict JSON sticky-note is *always* appended. For total stability, I also enabled Groq's native `response_format={"type": "json_object"}` setting to force JSON compliance at the system level.

## What This Does To My Project
The Demo Factory is no longer a manual script; it is a **closed-loop evaluation pipeline**. I can now feed the Brain a business problem, have it generate a testing architecture, and then hand that architecture to the Muscle (the Runner) to automatically execute, validate, and grade multiple AI models against real-world data without writing any new logic.
